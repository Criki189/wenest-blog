# Contributing to wenest-blog

Two ways content lands in this repo:

1. **Manual** — a human writes a markdown file and opens a PR.
2. **Automated** — n8n drafts an article with the Claude API, commits to a `drafts/<slug>` branch, and opens a PR for human review.

Either way, every article must pass `npm run validate-content` before merge. CI runs this automatically.

## Manual flow

```bash
npm run new-post -- --slug=spring-pool-prep-sydney --bucket=seasonal
# edit content/blog/spring-pool-prep-sydney.md
# drop a 1200x630 cover at public/images/blog/spring-pool-prep-sydney/cover.jpg
npm run validate-content
git checkout -b drafts/spring-pool-prep-sydney
git add . && git commit -m "Draft: Spring pool prep Sydney"
git push -u origin drafts/spring-pool-prep-sydney
gh pr create --title "Draft: Spring pool prep Sydney" --base main
```

A post is **only published** when, on `main`:

- `status: published`
- `published_date` is set
- branch is merged → Vercel redeploys

Posts with any other `status` are excluded from production builds, sitemap, and RSS. They render at their slug only when `NEXT_PUBLIC_SHOW_DRAFTS=1`.

## Automated flow (n8n → GitHub)

### Pipeline

```
┌─────────┐   ┌──────────┐   ┌─────────────┐   ┌─────────────────┐   ┌─────────┐
│  Cron   │──▶│ Claude   │──▶│ Unsplash +  │──▶│ GitHub Contents │──▶│  PR     │
│ /webhook│   │ (CLAUDE  │   │ DALL·E body │   │ API (commits)   │   │ opened  │
│         │   │  .md as  │   │ images      │   │                 │   │         │
│         │   │  system) │   │             │   │                 │   │         │
└─────────┘   └──────────┘   └─────────────┘   └─────────────────┘   └─────────┘
```

n8n must commit, on a single branch `drafts/<slug>`:

```
content/blog/<slug>.md                  status: "review"
public/images/blog/<slug>/cover.jpg
public/images/blog/<slug>/body-1.jpg    (if body_images[0])
public/images/blog/<slug>/body-2.jpg    (if body_images[1])
```

Then open a PR titled `Draft: <article title>` against `main`.

### GitHub Contents API payloads

All requests need `Authorization: Bearer <GITHUB_TOKEN>` and `Accept: application/vnd.github+json`. The token needs `contents:write` and `pull-requests:write` on this repo.

#### 1. Create the branch

```http
POST /repos/wenest/wenest-blog/git/refs
{
  "ref": "refs/heads/drafts/spring-pool-prep-sydney",
  "sha": "<sha of main HEAD, fetched from /git/refs/heads/main>"
}
```

#### 2. Commit the markdown file

```http
PUT /repos/wenest/wenest-blog/contents/content/blog/spring-pool-prep-sydney.md
{
  "message": "Draft: Spring pool prep Sydney (markdown)",
  "branch": "drafts/spring-pool-prep-sydney",
  "content": "<base64-encoded markdown including frontmatter>"
}
```

#### 3. Commit each image (one PUT per file)

```http
PUT /repos/wenest/wenest-blog/contents/public/images/blog/spring-pool-prep-sydney/cover.jpg
{
  "message": "Draft: Spring pool prep Sydney (cover image)",
  "branch": "drafts/spring-pool-prep-sydney",
  "content": "<base64-encoded JPEG bytes>"
}
```

Repeat for `body-1.jpg`, `body-2.jpg` if applicable.

#### 4. Open the PR

```http
POST /repos/wenest/wenest-blog/pulls
{
  "title": "Draft: Spring pool prep Sydney",
  "head": "drafts/spring-pool-prep-sydney",
  "base": "main",
  "body": "Auto-generated by n8n. Set status: published and published_date, then merge to publish."
}
```

### Frontmatter the LLM must produce

The system prompt for the Claude call should be the contents of `/CLAUDE.md`. The user prompt asks Claude to return YAML frontmatter + markdown body that satisfies the schema in `lib/content.ts` (`FrontmatterSchema`).

Required fields: `title`, `slug` (must equal filename), `description` (140-155 chars), `primary_keyword`, `bucket`, `status: "review"`, `target_word_count`, `cover_image: "/images/blog/<slug>/cover.jpg"`, `cover_image_alt`, `last_updated`. If `faq: true` then `faq_items` is required.

CI will reject the PR if `npm run validate-content` fails.

## Reviewing and publishing

Cristian reviews on GitHub. To publish:

1. Edit `content/blog/<slug>.md` in the PR's branch.
2. Set `status: "published"` and `published_date: "<today ISO>"`.
3. Bump `last_updated` if any edits were made.
4. Merge to `main`.
5. Vercel auto-redeploys; the article is live at `wenest.com.au/blog/<slug>`.

## Editorial standards

See `/CLAUDE.md`. Tl;dr: trust-first, Australian English, six content buckets, FAQ + soft CTA. No AI-generic phrasing ("In today's fast-paced world", "Let's dive in", "navigating the complexities of").
